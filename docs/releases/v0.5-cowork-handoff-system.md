# v0.5 Multi-Agent Cowork Handoff System

## Status

Local technical release candidate.

## Summary

v0.5 turns v0.4's agent/skill/environment configuration into an actual, ready-to-use execution handoff. `handoff generate` resolves a project's active feature, maps its `status.md` `current_state` to the recommended next agent (reusing the same state map proven in the runtime kit's coordinator), and composes a complete, copy-pasteable prompt covering the execution surface, environment, overall goal, this step's goal, agent profile, inputs, expected outputs, allowed paths, forbidden actions, validation commands, and the final report format the agent should follow. It does not execute anything: the orchestrator produces instructions, and a human or an external tool (Claude Cowork, Codex, Gemini, or plain copy/paste) carries them out.

## What changed from v0.4

- added a handoff data model (`Handoff` in `lib/handoff-types.ts`) covering project, feature, orchestration goal, agent, environment, context, expected outputs, allowed paths, forbidden actions, validation commands, and the composed prompt;
- added feature resolution (`lib/feature-resolver.ts`) that reuses the same feature-folder diagnostics `doctor`/`next`/`project inspect` already use, plus a minimal `status.md` frontmatter reader (`lib/status-doc.ts`) scoped to the orchestrator CLI (the runtime kit's own frontmatter parser is a template copied into target repos, not a dependency of this package);
- ported the proven state → agent/target-state/expected-outputs mapping from the runtime kit's coordinator (`scripts/agent/lib/state_map.ts`) into `lib/workflow-state-map.ts`, translated to the v0.4 config's agent naming;
- added `handoff generate`, `handoff write`, `handoff show`, and `handoff list` commands;
- `next` now additionally suggests `handoff generate <project>` once a project is `READY` or `CHECK_WARNINGS`, without removing or changing its existing recommendation;
- all v0.1–v0.4 commands are unchanged in behavior.

## New commands

```text
npm run agentic-sdd -- handoff generate <name-or-target-repo> [--agent <name>] [--feature <id>] [--environment <name>]
npm run agentic-sdd -- handoff write <name-or-target-repo> [--agent <name>] [--feature <id>] [--environment <name>]
npm run agentic-sdd -- handoff show <name-or-target-repo> [--feature <id>]
npm run agentic-sdd -- handoff list <name-or-target-repo>
```

### `handoff generate`

Resolves the project (registered name or direct path), validates the runtime kit is installed, resolves the active feature (auto-selects when exactly one exists; requires `--feature <id>` and lists candidates when more than one exists; fails with the exact `init-feature` command to run when none exists), reads `current_state` from that feature's `status.md`, maps it to a recommended agent/target-state/expected-outputs, resolves that agent's profile/skills and the requested (or default `local_windows_powershell`) environment, and prints a metadata header (project, feature, current/target state, agent, environment, estimated context file count) followed by the full prompt. Read-only: never writes to the target repository, never calls an AI model.

### `handoff write`

Runs the same generation, then writes it under the orchestrator's own local state at `.agentic-sdd/handoffs/<project-slug>/<feature-id>/<timestamp>/`:

- `handoff.json` — the full structured handoff record;
- `prompt.md` — the composed prompt, ready to paste;
- `context_files.txt` — one input file path per line.

`.agentic-sdd/` (already gitignored since v0.3) covers this new subdirectory automatically. Never writes into the target repository unless the target repository itself is where a feature's own files live (which `handoff write` does not touch — it only reads `status.md`).

### `handoff show`

Reads and prints the most recently written handoff for a project (optionally scoped to `--feature <id>`). If none has been written yet, it suggests `handoff write`.

### `handoff list`

Lists every previously written handoff for a project across all its features, newest first.

## What a handoff contains

Per the `Handoff` model (`version: 1`): `project` (name, path), `feature` (id, slug, path, current/target state), `orchestration` (overall goal, this-step goal, `safetyMode: "semi_assisted"`, `executionMode: "cowork"`), `agent` (name, role, provider, external reference, execution mode, profile, active skills), `environment` (name, shell, command style, execution surface, tools), `context` (input files, summary, context policy), `expectedOutputs`, `allowedPaths`, `forbiddenActions`, `validationCommands`, and the fully composed `prompt`.

The prompt itself is organized into eleven lettered sections (A–K): execution surface, available environment, overall orchestration goal, this execution goal, agent profile, inputs, expected outputs, allowed paths, forbidden actions, validation commands, and final report format.

## How to use with Claude Cowork

Run `handoff generate <project> --environment claude_cowork_browser` (or omit `--environment` and let the agent's default apply if it differs), copy the printed prompt, and paste it into a Claude Cowork session. The prompt explicitly states "You are running inside Claude Cowork in the browser" and instructs the agent not to assume local file or shell access beyond what is pasted.

## How to use with Codex/Gemini/manual copy-paste

The same flow works for any external tool: generate the handoff, copy the prompt, paste it into Codex, Gemini, or a plain manual copy-paste exchange (`--environment manual_copy_paste`). The agent, skills, and forbidden actions in the prompt reflect the `codex_architect`/`codex_reviewer`/`gemini_product_owner`/`gemini_validator` role recommended for the feature's current state, or whichever agent is passed via `--agent`.

## Safety guarantees (unchanged from v0.1–v0.4, extended)

- GitHub and `status.md` remain the source of truth; the handoff only reads `status.md`, never writes it.
- The orchestrator does not execute AI models. `handoff generate`/`write`/`show`/`list` are entirely local, read-only-against-the-target operations (the only write is `handoff write`'s own output under the orchestrator's `.agentic-sdd/handoffs/`).
- Every generated prompt's forbidden-actions section always includes: do not auto-merge, do not call external AI APIs, do not add secrets or credentials, do not delete user files, do not create a new branch unless explicitly instructed, do not scan the full repository unless blocked, do not create duplicate documentation files, do not modify unrelated workflow/runtime files unless required — plus the specific agent's own forbidden actions from its v0.4 config.
- No autonomous execution, no automatic merge, no dashboard, no API mode.
- Human review and final merge remain required; the prompt's final report format explicitly tells the agent not to merge or create a pull request itself.

## How this prepares v0.6 (Cycles, Budgets & Token Estimation)

The handoff's `context.inputFiles` list is exactly the input a future context-size/token estimator (v0.6) needs to size before generation; `expectedOutputs` and the state map's `targetStates` are the same data a future cycle/iteration-limit tracker would consume to detect repeated rejection loops. Neither is implemented in v0.5 - the handoff model intentionally carries the fields a v0.6 estimator/budget tracker will read, without yet computing or enforcing anything numeric.

## Limitations

- Feature-to-agent state mapping covers the seven states the runtime kit coordinator actively models (`IDEA_CAPTURED`, `BRANCH_INITIALIZED`, `READY_FOR_TECHNICAL_SPEC`, `READY_FOR_IMPLEMENTATION`, `IMPLEMENTATION_READY_FOR_REVIEW`, `READY_FOR_FUNCTIONAL_VALIDATION`, `READY_FOR_PR`); a feature in any other state (including blocked states) does not get an executable handoff, and `handoff generate` explains why instead of guessing.
- If `status.md`'s `current_state` cannot be read (missing file, missing frontmatter), the handoff generator reports `state_unknown` as an error rather than inventing a state, and recommends running `doctor`/checking `status.md` manually.
- No token/context size estimation yet (`context.inputFiles` lists paths only, not sizes) - that is v0.6 scope.
- `handoff write`'s history is local to the orchestrator's machine and gitignored, same as the v0.3 project registry and v0.4 config override.
- The default environment when `--environment` is omitted is always `local_windows_powershell`; there is no per-project or per-agent default environment selection yet.

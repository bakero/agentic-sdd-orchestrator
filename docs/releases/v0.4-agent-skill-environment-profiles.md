# v0.4 Agent, Skill & Environment Profiles

## Status

Local technical release candidate.

## Summary

v0.4 gives the orchestrator a formal, local, inspectable configuration model for the things a Cowork handoff always needed but only expressed informally: which agents exist, what role and skills each one has, whether it runs manually or automatically, which prompt profile it uses, and which execution environment (shell, tools, browser access) it is expected to run in. Five default agents, eighteen default skills, five default prompt profiles, and five default environments ship built in. A user can inspect all of it (`profile`, `agent`, `env`, `config show`), create a local editable copy (`config init`), and validate it (`config validate`) without any of it calling an AI model or executing anything automatically.

## What changed from v0.3

- added a versioned, typed configuration model (`packages/cli/src/config/`) covering agents, profiles, skills, and environments;
- added built-in defaults for 5 agents, 5 prompt profiles, 18 skills, and 5 environments;
- added `profile list` / `profile show`, `agent list` / `agent show`, `env list` / `env show`;
- added `config init`, `config show`, `config validate`;
- added `project config <name-or-path>` to preview which agents/environments would apply to a registered or direct-path project;
- added a resolution module (`lib/prompt-context.ts`) that resolves an agent's profile, skills, and prompt-guidance fragments together with an environment, as a seam for v0.5's handoff generator to build on;
- `agent:log-call` requirement (added in v0.3) and all existing `inspect`/`install`/`init-feature`/`project`/`doctor`/`next` commands are unchanged in behavior.

## New commands

```text
npm run agentic-sdd -- profile list
npm run agentic-sdd -- profile show <profile-name>
npm run agentic-sdd -- agent list
npm run agentic-sdd -- agent show <agent-name>
npm run agentic-sdd -- env list
npm run agentic-sdd -- env show <environment-name>
npm run agentic-sdd -- config init [--force]
npm run agentic-sdd -- config show
npm run agentic-sdd -- config validate
npm run agentic-sdd -- project config <name-or-target-repo>
```

### `profile list` / `profile show`

Lists the prompt profiles (functional discovery, technical specification, implementation, technical review, functional validation) and, for a given profile, its prompt template reference, context policy, recommended reasoning level, and which agent(s) use it by default.

### `agent list` / `agent show`

Lists the configured agents with role, default profile, execution mode, and provider. `agent show` additionally prints the agent's description, external tool reference (a Gemini Gem / Custom GPT / Claude Project name — a label only, never called by the orchestrator), active skills, responsibilities, and forbidden actions.

### `env list` / `env show`

Lists the configured execution environments with shell, execution surface, and available tools. `env show` additionally prints OS, command style, the full tool availability map, execution rules, forbidden actions, and expected output format.

### `config init`

Creates `.agentic-sdd/config.json` from the built-in defaults. Refuses to overwrite an existing file unless `--force` is passed. After creation, suggests `config validate` as the next step.

### `config show`

Prints the effective configuration (the local override if `.agentic-sdd/config.json` exists, otherwise the built-in defaults) along with a summary count and which source was used.

### `config validate`

Validates the effective configuration:

- duplicate agent/profile/skill/environment names;
- agent `defaultProfile` references an existing profile;
- agent `skills` reference existing skills;
- agent `executionMode` is `manual` or `automatic`;
- profile `contextPolicy` and `recommendedReasoning` are valid enum values;
- profile `promptTemplate` looks like a real template reference (a `.md` filename — a warning, not a hard failure, since this is a naming heuristic, not a filesystem check);
- environment has `os`, `shell`, `executionSurface`, and `tools`;
- the JSON itself is well-formed (a parse failure surfaces as a validation error, not a crash).

Exits non-zero when validation fails, matching the runtime kit validator's own convention.

### `project config`

Resolves a project by registered name or direct path (same resolution as `doctor`/`next`) and prints which agents/profiles/environments the orchestrator's effective config would apply to it. v0.4 has no per-project config override yet — every project currently sees the same orchestrator-level effective config — so this command is a preview of that shared config scoped to the project's identity, not yet a per-project customization surface. It never writes to the project or to the orchestrator config.

## Default agents

| Agent | Role | Default profile | Execution mode | Provider |
|---|---|---|---|---|
| `gemini_product_owner` | Product Owner | `functional_discovery` | manual | gemini |
| `codex_architect` | Technical Architect | `technical_specification` | manual | codex |
| `claude_implementer` | Implementer | `implementation` | manual | claude |
| `codex_reviewer` | Technical Reviewer | `technical_review` | manual | codex |
| `gemini_validator` | Functional Validator | `functional_validation` | manual | gemini |

Each agent definition also carries a description, an external tool reference (Gem/Custom GPT/Claude Project name, label only), a skill list, responsibilities, and forbidden actions — see `agent show <name>` for the full record.

## Default skills

`requirements_interview`, `acceptance_criteria`, `scope_control`, `one_question_at_a_time`, `repo_analysis`, `minimal_context_pack`, `technical_specification`, `test_plan_design`, `implementation_boundaries`, `code_implementation`, `test_execution`, `technical_review`, `functional_validation`, `cost_control`, `token_discipline`, `cowork_handoff_preparation`, `browser_execution_safety`, `powershell_command_safety`.

Each skill carries a short description, the agents it applies to, one line of prompt guidance, and the anti-patterns it exists to prevent — kept intentionally concise rather than full prompt essays.

## Default environments

| Environment | OS | Shell | Execution surface |
|---|---|---|---|
| `local_windows_powershell` | windows | powershell | local_shell |
| `claude_cowork_browser` | any | none | claude_cowork_browser |
| `manual_copy_paste` | any | none | manual_copy_paste |
| `local_macos_zsh` | macos | zsh | local_shell |
| `local_linux_bash` | linux | bash | local_shell |

Each environment declares its available tools (git/gh/node/npm/browser), execution rules, forbidden actions, and expected output format. `local_windows_powershell` explicitly forbids bash-only commands and requires PowerShell-compatible syntax; the two non-shell environments (`claude_cowork_browser`, `manual_copy_paste`) explicitly forbid assuming any local file, shell, or command-execution access.

## Execution modes

- **manual** (the default for every v0.4 agent): the human explicitly selects the provider/model/external tool for that step; the orchestrator preserves that choice and never overrides it.
- **automatic**: declared as a valid mode in the schema and validated, but v0.4 does not act on it — no API is called, no provider is auto-selected, and no execution happens. It exists as a stable place for a future recommendation-only mechanism to attach to; see the "How this prepares v0.5" section below.

## How this prepares v0.5 (Multi-Agent Cowork Handoff)

v0.4 deliberately stops short of generating a full Cowork handoff package. It adds the resolution seam v0.5 needs instead:

- `lib/config.ts` resolves an agent, profile, skill, or environment by name from the effective config;
- `lib/prompt-context.ts`'s `resolveAgentContext`/`resolveEnvironmentContext`/`resolveHandoffInputs` combine those lookups into what a handoff/prompt generator will need for one step — the agent definition, its profile, its resolved skills, and the flattened prompt-guidance fragments those skills contribute, alongside a resolved environment.

None of this renders a prompt or writes any file; it is read-only resolution logic that v0.5 can call into instead of re-deriving agent/skill/environment lookups from scratch.

## Configuration storage

Two layers, matching the project registry's pattern from v0.3:

- **Built-in defaults** — `packages/cli/src/config/defaults/default-config.json`, versioned in this repo.
- **Local override** — `.agentic-sdd/config.json`, gitignored, created by `config init`. When present, it is used as-is as the effective config (v0.4 does not deep-merge a partial override against defaults — `config init` writes a complete, edit-in-place file).

If no local override exists, every command falls back to the built-in defaults automatically.

## Limitations

- `config init` writes a full copy of the defaults; there is no partial/patch override format yet, and no per-project override (see `project config` above).
- `automatic` execution mode is declared and validated but has no effect in v0.4 — no provider is called or recommended programmatically yet.
- `promptTemplate` references are checked for a plausible `.md`-style name only; v0.4 does not verify the referenced template file actually exists on disk.
- No prompt is rendered or handoff package generated yet; that is v0.5 scope.

## Safety guarantees (unchanged from v0.1–v0.3)

- GitHub and `status.md` remain the source of truth.
- Cowork mode stays semi-assisted: the CLI generates configuration, diagnostics, and (starting v0.5) prompts; a human or tool executes them.
- No external AI API is called by `profile`, `agent`, `env`, `config`, or `project config`.
- No dashboard, hosted control plane, or API mode is introduced.
- No autonomous agent execution or automatic merge is performed. `automatic` execution mode is a declared setting only.
- `config`/`profile`/`agent`/`env`/`project config` are strictly read-only against target repositories; `config init` only ever writes inside the orchestrator's own `.agentic-sdd/` directory.
- Human review and final merge remain required.

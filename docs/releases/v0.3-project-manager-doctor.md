# v0.3 Project Manager and Doctor

## Status

Local technical release candidate.

## Summary

v0.3 makes the orchestrator easier to use across multiple target repositories. It adds a local project registry so a target repository can be referred to by a short name instead of a full path, a `doctor` command that diagnoses what is missing or inconsistent in a target repository, and a `next` command that recommends the single most important next safe action. None of these commands execute anything on the user's behalf, call an external AI API, or change the existing semi-assisted safety model.

## What changed from v0.2

- added a local project registry (`.agentic-sdd/projects.json` inside the orchestrator repo, gitignored) that maps a short project name to an absolute target repository path;
- added a `project` command group: `add`, `list`, `remove`, `inspect`;
- added a `doctor` command that runs a fuller set of PASS/WARN/FAIL checks against a target repository and prints an overall status plus a recommended next command;
- added a `next` command that reuses the same diagnostics as `doctor` but prints only the single most important next safe action;
- `doctor`/`next`/`project inspect` accept either a registered project name or a direct filesystem path, so named projects are not required to use the tool;
- the runtime kit's required package scripts now also include `agent:log-call` (previously only checked implicitly), so `install` and `doctor` agree on what "fully set up" means;
- the existing `inspect`, `install`, and `init-feature` commands are unchanged in behavior and continue to work exactly as documented in v0.2.

## New commands

```text
npm run agentic-sdd -- project add <target-repo> --name <name>
npm run agentic-sdd -- project list
npm run agentic-sdd -- project remove <name>
npm run agentic-sdd -- project inspect <name-or-target-repo>
npm run agentic-sdd -- doctor <name-or-target-repo>
npm run agentic-sdd -- next <name-or-target-repo>
```

### `project add`

Registers a local target repository under a short name. Validates that:

- `--name` is present and not already registered;
- the target path exists;
- the target path is a git repository (`.git` present).

Writes `.agentic-sdd/projects.json` (creating `.agentic-sdd/` if needed) with an absolute, normalized path, then prints the recommended next command (`doctor <name>`).

### `project list`

Lists registered project names and paths. If none are registered, it says so and suggests `project add`.

### `project remove`

Removes a project from the registry by name. It never deletes or modifies files in the target repository, and it does not ask for interactive confirmation.

### `project inspect`

Resolves a registered name (or accepts a direct path as a fallback) and reports: git repository status, whether the runtime kit is installed, whether `package.json` is present, whether the required agent npm scripts are present, whether `.agent_runtime/` exists, whether `docs/features` exists, and any detected feature folders.

### `doctor`

Runs a read-only diagnostic sweep against a target repository (by registered name or direct path) and prints PASS/WARN/FAIL for each check:

- target path exists and is a directory;
- target is a git repository;
- `package.json` exists;
- runtime kit folders exist (`.agents/`, `docs/agents/`, `docs/features/`, `docs/metrics/`, `scripts/agent/`);
- `.github/workflows/agent-sdd-guard.yml` exists;
- required `package.json` agent scripts exist (`agent:validate`, `agent:next`, `agent:prompt`, `agent:log-call`);
- `.agent_runtime/` is listed in `.gitignore` (only checked once `.agent_runtime/` exists);
- at least one `docs/features/<issue>-<slug>/` folder exists, and each one has a `status.md`;
- `node_modules` exists (dependencies likely installed);
- `.agent_runtime/next_prompt.md` exists (a prompt has been generated).

At the end it prints an overall status (`READY`, `NEEDS_SETUP`, `NEEDS_FEATURE`, `NEEDS_NEXT_PROMPT`, or `CHECK_WARNINGS`) and a recommended next command. `doctor` never writes to the target repository.

### `next`

Runs the same diagnostics as `doctor` and prints only the single most important next safe action (a one-line summary, the command to run if there is one, and why). It does not execute that action, call an AI model, or modify any files. It exists to reduce decision friction when a user does not want the full `doctor` report.

## Supported flow

```text
1. npm run agentic-sdd -- project add <target-repo> --name <name>
2. npm run agentic-sdd -- doctor <name>
3. npm run agentic-sdd -- next <name>              (repeat as you follow each recommendation)
4. npm run agentic-sdd -- install <target-repo>
5. npm run agentic-sdd -- init-feature <target-repo> --issue <number> --slug <slug> --title <title>
6. cd <target-repo>
7. npm install
8. npm run agent:next
9. Open .agent_runtime/next_prompt.md
```

`doctor`/`next` can be re-run after each step to confirm progress; the overall status advances `NEEDS_SETUP` -> `NEEDS_FEATURE` -> `NEEDS_NEXT_PROMPT` -> `READY` (or `CHECK_WARNINGS` if only non-blocking warnings remain).

## Examples (PowerShell)

```powershell
$sandbox = Join-Path $env:TEMP "agentic-sdd-v03-demo"
Remove-Item -Recurse -Force $sandbox -ErrorAction SilentlyContinue
mkdir $sandbox | Out-Null
git init $sandbox

npm run agentic-sdd -- project add $sandbox --name demo
npm run agentic-sdd -- project list
npm run agentic-sdd -- project inspect demo
npm run agentic-sdd -- doctor demo
npm run agentic-sdd -- next demo
npm run agentic-sdd -- install $sandbox
npm run agentic-sdd -- doctor demo
npm run agentic-sdd -- init-feature $sandbox --issue 1 --slug demo-feature --title "Demo feature"
npm run agentic-sdd -- doctor demo

cd $sandbox
npm install
npm run agent:next

cd -
npm run agentic-sdd -- doctor demo
npm run agentic-sdd -- next demo
```

## Non-goals

- no dashboard;
- no API mode;
- no external AI API calls;
- no autonomous agent execution;
- no automatic merge;
- no GitHub App behavior;
- no remote repository cloning;
- no global npm install requirement.

## Limitations

- the project registry is local to the orchestrator repository and machine (`.agentic-sdd/projects.json`, gitignored) — it is not shared or synced;
- `doctor`/`next` diagnostics are structural (file/folder/script presence), not semantic — they do not validate document content;
- `project remove` does not detect or warn about dangling references elsewhere; it only edits the registry;
- registering a project does not install anything; `install` is still a separate, explicit step;
- see [known limitations](../product/known-limitations.md) for the full list carried over from v0.1/v0.2.

## Safety guarantees (unchanged from v0.1/v0.2)

- GitHub and `status.md` remain the source of truth.
- Cowork mode stays semi-assisted: the CLI generates prompts and diagnostics, but a human or tool executes/acts on them.
- No external AI APIs are called by the orchestrator.
- No dashboard, hosted control plane, or API mode is introduced.
- No autonomous agent execution or automatic merge is performed by `doctor`, `next`, or any `project` command.
- `doctor` and `next` are strictly read-only against the target repository.
- Human review and final merge remain required.
